这是一个非常好的用户体验优化点！对于“懒人模式”的语音输入，**静音检测（Silence Detection）** 是标配功能。

### 交互设计
1.  **触发条件**：当用户开始录音后，系统实时监测。
2.  **静音判定**：如果用户停止说话超过 **2.5秒**（这是一个比较舒适的阈值，既给了用户思考的间隙，又不至于让等待时间过长）。
3.  **自动动作**：系统自动执行“停止录音”操作，并无缝衔接进入“AI 分析”流程。
4.  **视觉反馈**：在界面上可以保留“正在聆听...”的提示，用户不需要做任何额外操作。

### 技术实现方案
我们需要引入一个**防抖（Debounce）计时器**来监测语音输入的间歇。

1.  **监听 `tempTranscript` 变化**：
    *   `tempTranscript` 是实时语音转录的临时结果。只要用户在说话，这个值就会不断变化（或者至少 Web Speech API / Native Plugin 会触发回调）。
    *   我们需要一个 `useEffect` 来监听 `tempTranscript` 的变化。

2.  **计时器逻辑**：
    *   每当 `tempTranscript` 更新（或者收到新的语音包）时，清除旧的计时器，启动一个新的计时器（例如 2.5s）。
    *   如果计时器触发了，说明用户已经 2.5s 没说话了。
    *   此时调用 `toggleListening()` 来停止录音。

3.  **细节处理**：
    *   **Native 端**：Native 插件通常通过 `partialResults` 事件返回数据。我们需要确保在 `partialResults` 回调里重置计时器。
    *   **Web 端**：Web Speech API 的 `onresult` 会频繁触发，适合用来重置计时器。
    *   **空状态处理**：如果用户一开始就没说话，`tempTranscript` 是空的，这时候不应该触发自动停止（或者应该触发“未检测到语音”的提示）。我们可以设定只有当 `importText` 或 `tempTranscript` **有内容**时，才启动这个静音检测。
