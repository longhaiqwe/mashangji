这是一个浏览器的兼容性检查逻辑过于严格的问题。

### 问题分析
1.  代码目前使用 `!Capacitor.isNativePlatform() && typeof window !== 'undefined' && 'webkitSpeechRecognition' in window` 来判断是否支持 Web 语音 API。
2.  Chrome 桌面版和移动版都支持 `webkitSpeechRecognition`，但用户可能使用了非标准环境或某些 Chrome 版本行为不一致。
3.  最关键的是：`recognition` 状态的检查逻辑在 `toggleListening` 中是：
    ```typescript
    if (!recognition) {
      alert('您的浏览器不支持语音输入...');
      return;
    }
    ```
    而 `recognition` 实例是在 `useEffect` 中初始化的。如果 `useEffect` 还没执行完或者初始化条件没满足（例如 `window` 对象还没准备好），`recognition` 就可能是 `null`。

### 修复计划
1.  **放宽 Web Speech API 检测**：不仅检查 `webkitSpeechRecognition`，也检查标准的 `SpeechRecognition`（虽然 Chrome 主要用前者，但为了兼容性最好都查）。
2.  **优化初始化逻辑**：确保在组件挂载时正确初始化 `recognition` 对象。
3.  **修改 `toggleListening` 中的报错提示**：如果是 Web 环境且没有 `recognition`，尝试**即时初始化**一次，而不是直接报错。

### 具体实施
1.  修改 `useEffect` 中的初始化代码，支持 `window.SpeechRecognition || window.webkitSpeechRecognition`。
2.  在 `toggleListening` 中，如果 `recognition` 为空，尝试再次获取一次构造函数。
